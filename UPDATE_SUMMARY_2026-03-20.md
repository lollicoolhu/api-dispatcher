# 2026-03-20 更新汇总

## 概述

今天完成了多个重要功能的开发和优化，主要集中在临时修改的条件匹配、日志展示和用户体验改进方面。

## 更新列表

### 1. 根级别数组匹配支持 ✓

**问题**: 当 POST 请求体本身是数组时（如 `[{...}, {...}]`），无法正确匹配数组元素。

**解决方案**:
- 支持使用 `[0].property` 访问根数组的第一个元素
- 支持使用 `[*].property` 访问根数组所有元素
- 向后兼容原有的对象路径匹配

**文档**: [ROOT_ARRAY_FIX.md](./ROOT_ARRAY_FIX.md)

**测试**: 28个测试用例全部通过

---

### 2. 通配符条件匹配逻辑优化 ✓

**问题**: 使用通配符 `[*]` 时，`eq` 操作符比较整个数组的 JSON 字符串，无法方便地检查"数组中是否包含某个值"。

**解决方案**:
- `[*].category = 2` 现在表示：检查数组中是否有任意元素的 category 等于 2
- 支持组合条件：`[*].category = 1` AND `[*].category = 2` 可以匹配同时包含两个值的数组
- 自动处理数字和字符串的类型转换

**文档**: [WILDCARD_MATCHING_UPDATE.md](./WILDCARD_MATCHING_UPDATE.md)

**测试**: 35个测试用例全部通过（包括7个新的条件匹配测试）

**示例**:
```json
// 请求体
[
  { "category": 2, "param": 0.05 },
  { "category": 1, "param": 5 }
]

// 条件配置
// 条件1: [*].category = 1
// 条件2: [*].category = 2
// 逻辑: AND
// 结果: ✓ 匹配
```

### 3. 日志中显示临时修改匹配信息 ✓

**功能**: 在访问日志详情中显示临时修改命中了哪条条件。

**显示内容**:
- 版本名称（优先使用备注）
- 匹配类型（条件匹配/无条件兜底）
- 条件逻辑（且/或）
- 匹配条件列表（来源.键 操作符 值）

**文档**: [MATCH_INFO_LOGGING.md](./MATCH_INFO_LOGGING.md)

**效果**:

**功能**: 在日志中点击版本名称，弹窗高亮展示并可修改该版本。

**特性**:
- 版本名称显示为蓝色下划线链接
- 点击后自动打开临时修改管理弹窗
- 自动展开并高亮命中的版本
- 平滑滚动到目标版本
- 持续高亮显示，直到关闭弹窗

**文档**: [CLICKABLE_VERSION_FEATURE.md](./CLICKABLE_VERSION_FEATURE.md)

## 使用建议

### 1. 配置临时修改版本时

- 为每个版本添加有意义的备注
- 使用清晰的条件配置
- 合理设置优先级

### 2. 使用路径匹配时

- 根级别数组：使用 `[0].property`
- 对象中的数组：使用 `items[0].property`
- 通配符匹配：使用 `[*].property` 检查任意元素

### 3. 调试条件匹配时

- 查看日志详情中的"临时修改匹配信息"
- 点击版本名称快速查看和编辑版本
- 使用通配符 + AND 逻辑匹配多个值
