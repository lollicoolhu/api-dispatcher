# 根级别数组匹配功能修复

## 问题描述

当 POST 请求的 body 本身是一个数组时（例如 `[{...}, {...}]`），之前的路径匹配功能无法正确访问数组元素。

### 问题示例

**请求配置:**
- 路径: `/api/v2/report/shareImages`
- 条件: `item[0].category = 2`

**实际请求 Body:**
```json
[
  { "category": 2 },
  { "category": 1 }
]
```

**问题:** 条件无法匹配，因为路径 `item[0].category` 期望的数据结构是：
```json
{
  "item": [
    { "category": 2 }
  ]
}
```

但实际的请求体根就是数组，没有 `item` 这个键。

## 解决方案

### 修改内容

1. **更新 `getValueByPath()` 函数** (`lib/request-handler.js`)
   - 改进了对根级别数组的处理
   - 现在支持以 `[0]` 开头的路径来访问根数组元素

2. **更新文档** (`ARRAY_MATCHING_GUIDE.md`)
   - 添加了"根级别数组访问"章节
   - 添加了根级别数组的使用示例
   - 更新了常见问题部分

3. **更新测试** (`test-path-matching.js`)
   - 添加了 9 个根级别数组测试用例
   - 所有 28 个测试用例全部通过

### 正确的配置方式

对于上述问题，正确的配置应该是：

**条件配置:**
- 来源: `body`
- 键: `[0].category` （而不是 `item[0].category`）
- 操作: `eq` (等于)
- 值: `2`

## 支持的根级别数组路径格式

```
[0]              # 访问根数组的第一个元素
[0].category     # 访问根数组第一个元素的 category 属性
[1].name         # 访问根数组第二个元素的 name 属性
[*].id           # 获取根数组所有元素的 id（返回数组）
```

## 向后兼容性

此修复完全向后兼容，不影响现有的路径匹配功能：
- 简单键访问：`name`
- 嵌套对象：`user.name`
- 对象中的数组：`items[0].id`
- 通配符：`items[*].name`

## 使用建议

1. **判断请求体类型**: 先确认请求体是对象还是数组
2. **对象类型**: 使用 `property.subproperty` 或 `array[0].property`
3. **数组类型**: 使用 `[0].property` 直接访问数组元素
4. **调试技巧**: 在浏览器开发者工具中查看实际的请求 Body 结构
